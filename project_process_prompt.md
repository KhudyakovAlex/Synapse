# Порядок обработки репозитория

Папка с репозиторием должна обрабатываться строго в следующем порядке:
1. Обработка markdown-файлов.
2. Конвертация содержимого markdown-файлов в другие файлы.
3. Обновление сайта проекта в папке INDEX.
4. Загрузка репозитория на GitHub.

---

## 1. Обработка markdown-файлов

Перед заливкой и конвертацией в HTML исправь орфографические и пунктуационные ошибки во всех markdown-файлах. 
Отформатируй их. Не надо ничего переформулировать и добавлять от себя. Не надо расшифровывать аббревиатуры.
Исправь нумерацию всех пунктов и подпунктов во всех документах. Порядок этих пунктов в тексте менять не надо.

### Ссылки на таблицы базы данных

**ВАЖНО**: В документах при упоминании таблиц БД необходимо:
- В документах приложения (SynapsePDS_APP*.md) — делать ссылки на таблицы из `SynapsePDS_APP_DB.md`
- В документах прошивки (SynapsePDS_FW*.md) — делать ссылки на таблицы из `SynapsePDS_FW_DB.md`

**Формат**: названия таблиц выделять жирным (конвертер автоматически создаст ссылки на HTML)
- Пример: `**LUMINAIRES**`

Якорь формируется из номера и названия таблицы в нижнем регистре через дефис (например, `#35-luminaires`).

### Файлы для обработки

## 2. Конвертация содержимого markdown-файлов в другие файлы

### Скрипты конвертации

#### `INDEX/convert_to_html.py`
Автоматический конвертер MD → HTML
- **НЕ ИЗМЕНЯТЬ** без необходимости
- Поддерживает все текущие возможности
- При изменении MD-файлов — **ВСЕГДА запускать**
- Автоматически вызывает `update_diagrams.py` для обновления диаграмм

**Запуск:**
```bash
cd D:\Git\Synapse\INDEX
python convert_to_html.py
```

**Скрипт автоматически:**
- Найдёт все *.md файлы в папках PRD/ и PDS/
- Сконвертирует их в HTML с единым стилем
- Сохранит в INDEX/PRD/ и INDEX/PDS/
- Применит все правила форматирования:
  - Заголовки → `<h1>`, `<h2>`, `<h3>`, `<h4>`
  - Списки → `<ul>`, `<ol>`
  - Код → `<pre><code>`
  - Mermaid диаграммы → `<div class="mermaid">`
  - Два пробела в конце строки → `<br>`
  - Bold → `<strong>`, Italic → `<em>`
  - URL-ссылки (http://, https://) → `<a href="..." target="_blank">`

#### `INDEX/update_diagrams.py`
Автоматическое обновление диаграмм в INDEX/index.html
- Читает `Project/process.md`
- Обновляет содержимое `<div class="mermaid">` в `INDEX/index.html`
- Вызывается автоматически из `convert_to_html.py`
- Можно запускать отдельно: `python update_diagrams.py`

#### `Project/update_dates.py`
Автоматическое обновление дат в MD-документах
- Находит все изменённые MD-файлы (PRD/, PDS/, MOBILE/), добавленные в git индекс
- Обновляет строку "**Последнее изменение:**" на текущую дату
- Добавляет обновлённые файлы обратно в индекс
- **Запускать ПОСЛЕ `git add` и ПЕРЕД `git commit`**
- Обновляет даты ТОЛЬКО в изменённых файлах, не трогает неизменённые

**Команда:**
```bash
python Project/update_dates.py
```

### Конвертация документов
- Все Markdown-файлы (*.md) должны иметь HTML-вариант
- Сохранять форматирование: заголовки, списки, таблицы, блоки кода
- Конвертировать Mermaid диаграммы в интерактивный вид
- Для каждой Mermaid диаграммы создавать отдельную страницу (например, `db_schema_mermaid_diagram_0.html`)
- **Исключение**: диаграммы типа `sequenceDiagram` **НЕ ТРЕБУЮТ** отдельных страниц для полноэкранного просмотра — они достаточно компактны и читаемы на странице документа
- Сохранять относительные ссылки между документами

### Раздел "Текущий статус" на главной странице
- **⚠️ ВАЖНО**: Раздел "Текущий статус" на главной странице `INDEX/index.html` берётся из файла `Project/status.md`
- **⚠️ ОБЯЗАТЕЛЬНО**: При изменении `Project/status.md` нужно перед заливкой сайта на GFitHub обновить соответствующий раздел в `INDEX/index.html`
- Конвертер `convert_to_html.py` **НЕ обрабатывает** файл `Project/status.md` автоматически
- Заголовок раздела на странице = главный заголовок (h1) из файла `Project/status.md`
- В разделе должны быть ссылки на соответствующие страницы документации:
  - "Приложение. База данных" → `PDS/SynapsePDS_APP_DB.html`
  - "USML" → `PDS/SynapsePDS_USML.html`
  - "Bluetooth-соединение" → `PDS/SynapsePDS_Bluetooth.html`
  - И другие спецификации по аналогии

### Схема базы данных (Mermaid)
- **⚠️ ВАЖНО**: Mermaid-схема БД хранится ТОЛЬКО в файле `PDS/SynapsePDS_DB_scheme.md`
- **⚠️ НЕ ДОБАВЛЯТЬ** схему в файл `PDS/SynapsePDS_APP_DB.md`!
- **Автоматическая вставка**: Конвертер `convert_to_html.py` автоматически вставляет схему из `SynapsePDS_DB_scheme.md` на веб-страницу `INDEX/PDS/SynapsePDS_APP_DB.html`
- **Формат полей**: `тип_данных ИМЯ_ПОЛЯ PK/FK "комментарий"`
- **Пример**: `int ID PK "ID группы"`
- **Порядок**: тип данных ПЕРЕД именем поля (требование синтаксиса Mermaid)
- **⚠️ КРИТИЧЕСКИ ВАЖНО**: При добавлении, удалении или изменении таблиц в `PDS/SynapsePDS_APP_DB.md` **ОБЯЗАТЕЛЬНО** обновлять схему в `PDS/SynapsePDS_DB_scheme.md`:
  - Добавлять новые таблицы в блок `erDiagram`
  - Добавлять связи между таблицами (отношения `||--o{`, `||--||` и т.д.)
  - Обновлять поля в существующих таблицах
  - Обновлять связи при изменении внешних ключей
  - После изменений запускать `convert_to_html.py` для обновления HTML-страницы

### Страницы с иконками (SVG)
- **Файлы с иконками**:
  - `PDS/SynapsePDS_Icons_Controllers.md` → SVG из `MOBILE/Images/Ico/Controller/`
  - `PDS/SynapsePDS_Icons_Luminaires.md` → SVG из `MOBILE/Images/Ico/Luminaire/`
  - `PDS/SynapsePDS_Icons_Locations.md` → SVG из `MOBILE/Images/Ico/Location/`
- **Автоматическая вставка**: Конвертер `convert_to_html.py` автоматически вставляет SVG-иконки в HTML-страницы
- **Формат в MD-файле**: `XXX_nazvanie.svg  - Описание иконки`
- **Результат в HTML**: перед текстом вставляется `<img src="..." width="64" height="64">`
- **Если иконка не найдена**: HTML-код всё равно вставляется (браузер покажет, что файл отсутствует)
- **Точное соответствие имён**: имя файла в MD должно точно совпадать с именем SVG-файла

### Интерактивность диаграмм Mermaid

- **Превью**: диаграмма отображается на странице документа
- **Клик**: открывает диаграмму на отдельной странице в полном размере
- **Зум**: колесо мыши увеличивает/уменьшает (0.5x - 5x)
- **Панорамирование**: перетаскивание мышью (без зажатия клавиш)
- **Выделение текста**: Shift + выделение мышью (для копирования названий таблиц, полей и т.д.)
- **Точка зума**: левый верхний угол (диаграмма расширяется вправо и вниз)
- **Сброс выделения**: при новом клике с Shift старое выделение сбрасывается
- **Выделение сохраняется**: после отпускания Shift

#### Проверка исходных файлов перед коммитом

**ПЕРЕД КАЖДЫМ КОММИТОМ:**

1. **Проверить изменения в исходных файлах**:
   ```bash

   ```
   
2. **Если изменены файлы, из которых генерируется сайт**:

## Технические замечания для ИИ-ассистента

### Автоматическая обработка репозитория

- **КРИТИЧЕСКИ ВАЖНО**: При обработке репозитория согласно инструкциям (команда "Обработать папку репозитория согласно инструкциям в project_process_prompt.md") **ОБЯЗАТЕЛЬНО** выполнять все 5 шагов **ДО КОНЦА** без остановок и ожидания подтверждения:
  1. Обработка markdown-файлов (исправление ошибок, форматирование)
  2. Конвертация MD → HTML (`python INDEX/convert_to_html.py`)
  3. Обновление дат: `git add .` затем `python Project/update_dates.py`
  4. Обновление сайта в INDEX (уже сделано на шаге 2)
  5. **Автоматическая загрузка на GitHub** (`git add .`, `git commit`, `git push`)
- После выполнения шагов 1-4 **НЕМЕДЛЕННО** выполнять шаг 5 без запроса подтверждения у пользователя
- **НЕ ОСТАНАВЛИВАТЬСЯ** и **НЕ ЖДАТЬ** подтверждения пользователя — выполнять все шаги последовательно до полного завершения
- Использовать осмысленное сообщение коммита на основе выполненных изменений
- **Правило**: коммитить до конца все сам, не надо ждать пользователя
- **КРИТИЧЕСКИ ВАЖНО**: При изменении `PDS/SynapsePDS_APP_DB.md` **ОБЯЗАТЕЛЬНО** проверять и синхронизировать схему в `PDS/SynapsePDS_DB_scheme.md` — это часть шага 1 (обработка markdown-файлов)
- **КРИТИЧЕСКИ ВАЖНО**: При любом изменении структуры БД (добавление/удаление полей, изменение размеров, изменение порядка таблиц или полей) **ОБЯЗАТЕЛЬНО** увеличить версию `VERSION` в JSON-схеме в файле `PDS/SynapsePDS_USML.md` (раздел 3.3)
- **НОВОЕ ПРАВИЛО**: После выполнения всех шагов обязательной обработки ассистент обязан выдать пользователю **один PowerShell-скрипт** (одна строка с командами, разделёнными `;`), который повторяет все необходимые команды конвейера (convert_to_html.py → git add → Project/update_dates.py → git commit → git push) так, чтобы пользователь мог запустить его у себя и полностью воспроизвести процесс

### PowerShell в Windows

- **НЕ использовать `&&`** для объединения команд — PowerShell не поддерживает этот синтаксис
- Использовать `;` вместо `&&`: `cd D:\Git\Synapse; git status`
- Или выполнять команды по отдельности

### Терминал в Cursor

- **КРИТИЧЕСКИ ВАЖНО**: Если команда завершилась и появился промпт `PS D:\...>` или сообщение `[master ххххххх]`, значит команда выполнена УСПЕШНО
- **НЕМЕДЛЕННО ПРОДОЛЖАТЬ** следующую команду — НЕ ЖДАТЬ никаких дополнительных сигналов от системы
- Cursor часто НЕ определяет завершение команды автоматически — это известная системная проблема
- **ИГНОРИРОВАТЬ** сообщение "Command was canceled by the user" — если видишь промпт или результат команды, команда завершена
- **Ориентироваться ТОЛЬКО на вывод терминала**: промпт `PS D:\...>` или `[master ххххххх]` = команда завершена
- **НИКОГДА НЕ ЖДАТЬ** подтверждения от системы — если видишь результат команды, СРАЗУ выполняй следующую
- **НЕ ЗАСТАВЛЯТЬ пользователя отменять команды** — это ошибка ассистента, а не пользователя
- После `git commit` с сообщением `[master ххххххх]` — СРАЗУ делать `git push` в следующем вызове, не ждать
- **Правило**: если видишь промпт или результат команды → команда завершена → сразу следующая команда
- **ВАЖНО**: Если возникли проблемы с выводом терминала (команды выполняются с exit code 0, но вывод не отображается), в конце работы **ОБЯЗАТЕЛЬНО** выдать пользователю ОДИН PowerShell-скрипт для ручного запуска всего процесса (одна строка, команды через `;`). Пример: `cd D:\Git\Synapse\INDEX; python convert_to_html.py; cd ..; git add -A; python Project/update_dates.py; git commit -m "Update documentation"; git push`
- **КРИТИЧЕСКИ ВАЖНО**: Скрипт для ручного запуска ВСЕГДА должен быть ПОСЛЕДНИМ элементом в отчете ассистента, после всех описаний выполненных действий. Это обязательное требование для каждой обработки репозитория.

### Скрипт update_dates.py

- Если выводит "Нет измененных MD-файлов для обновления" — это нормально, скрипт завершён
- Скрипт обновляет даты только в файлах PRD/, PDS/, MOBILE/, добавленных в git индекс
- Файлы вне этих папок (например, project_process_prompt.md) не обрабатываются
